Introduction

In this lab, you will build on your knowledge of programming in Flutter by writing applications that permit asynchronous calls to avoid the UI from hanging. As you know Dart is a single threaded language and therefore execution of code would be frozen if a call was made that needed time to compute. This is where asynchronous methods can be used to avoid the program freeezing.

Recap

  • Async - declaring a function asynchronous means you are informing the compiler that the function needs to wait for a response

  • Await - means the application must wait here until the function is finished and you will get the return value

  • Future - is a type that comes from the future and returns the value from the asynchronous function. It completes with success (.then) or with error (.catchError) methods

  • .Then((value){...} - is the callback that is called once the future is completed successfully with a value

The difference between .then and await approach has the same end goal (i.e., the output is printed correctly). However, the await function tells your program to wait until the async function finishes.

xxxxxxxxxx
void main() async {
    await waitExample();
    print('Waiting');
}
Future waitExample() async {
    print('Called and waiting');
    return Future.delayed(Duration(seconds: 5), () {
        // waits for 5 seconds
        print("Done!");
    });
}

The output:

Called and waiting Done! Waiting

Whereas,

xxxxxxxxxx
void main() async {
    waitExample().then((_) {
        print("I'm really done!");
    });
    print('Waiting');
}
Future waitForMe() async {
    print('Called and waiting.');
    return Future.delayed(Duration(seconds: 3), () {
        print("Done!");
    });
}

The output:

Called and waiting Waiting Done! I'm really done!

 

 

You should use async when you want to define an async method. Followed by await is used when you need to wait for a method to finish and proceed with the rest of the program. Then (then) should be used when you want to process the Future after it was successfully finished in an asynchronous way (i.e., the program will continue to execute after the async call was made and then use the then callback later). Finally, a Future should be used when you want to get a result from an async function. These are needed when a normal function returns a result that happens in the Future. The Future keyword is used to inform the function when the result is ready.

class FutureBuilderExample extends StatefulWidget {

  @override
  FutureBuilderExampleState createState() => FutureBuilderExampleState();

class FutureBuilderExampleState extends State<FutureBuilderExample> {
  Future<String> fetchData() async {
    // Simulate fetching data from an API
    await Future.delayed(Duration(seconds: 2));
    return "Hello, Future Builder!";
  }


  @override
  Widget build(BuildContext context) {
    return Scaffold(
      appBar: AppBar(
        title: Text('FutureBuilder Example'),
      ),
      body: Center(
        child: FutureBuilder<String>(
          future: fetchData(),
          builder: (context, snapshot) {
            switch (snapshot.connectionState) {
              case ConnectionState.none:
                return Text('Press button to start.');
              case ConnectionState.active:
              case ConnectionState.waiting:
                return CircularProgressIndicator();
              case ConnectionState.done:
                if (snapshot.hasError)
                  return Text('Error: ${snapshot.error}');
                return Text('Data: ${snapshot.data}');
            }
          },
        ),
      ),
    );
  }
}

This snippet of code simulates the fetching of data from an API. Where it returns a hardcoded string. Inside the build() a FutureBuilder Widget handles the asynchronous data retrieval. The future property is set to fetchData() which contains the returned String from the method. The builder has four connection states (none, active, waiting and done).

It is not uncommon for mobile applications to make an HTTP request. This exchange of information is how mobile applications communicate to retrieve and send data. In this part of the lab, we will look at how we can retrieve information from the server. This is the process in which the application calls an endpoint (URL) on the Internet and retrieves data. The data is often called from an additional source such as retrieving information from a database, returning it via the RESTful call to the mobile application. But this could also be from an external endpoint.

First, we need to understand what an API call is. An Application Programming Interface (API) is a process that a server runs to read and write data in a controlled way. This allows the server to determine what a call can retrieve, how it’s retrieved and who can access the data. This is usually achieved by first making the ‘handshake’ you provide a username, password/token (we tend to use tokens more these days, based on the fact you aren’t sending a password over HTTP as it’s not secure) to the API. The server then authenticates if the provided access token grants access the data. If we think about Facebook again, third party developers can build apps that use Facebook’s infrastructure in some way. This could be as a Facebook login – where you don’t have your own account infrastructure for your project but instead use Facebook’s to handle all account information, password resets etc) and instead you have a token provided to you via Facebook. The token is the key that authenticates a user, you might also want to post a message to Facebook when you users unlock something or retrieve their contacts etc. All these features are handled by Facebook you have to be granted permission by the user but once this is achieved you can then make Facebook API calls on their behalf. Your users will then get the content/action they need from your application. It’s a little complicated but for now all you need to understand is that third party applications can retrieve and store data via API calls.

An API request comes in many forms; GET, POST, DELETE, PUT and PATCH. When making a HTTP call, we need to include the HTTP package in our Flutter application setup.

  • Packages

Including libraries in your Flutter app is a great way to extend the functionality of your app by utilising libraries that other developers have created. There are four stages to adding a package to your application.

  • Find the package you want to include - https://pub.dev/flutter/packages

  • Add it to the pubspec.yaml file

  • Import the package into the dart file you want to access the package

  • Use it within your code

Be careful when selecting a package to implement inside your application. You will see which platforms a particular package supports. In the example, a URL_launcher package is a Flutter package supporting all three platforms (Android, iOS and Web). Some packages won’t always support the platform you are targeting.

With packages you don’t download from the pub.dev site, all you need to do is include the package name and version in the pubspec.yaml file under the dependencies section (Figure 9). Sometimes you will notice the carat (^), this tells the pub 'get command' to use the latest version within a major version. I would always suggest you find out what version works and be specific in your package inclusion.

In order to ‘get’ the package we run the following command ‘flutter pub get’. This will then download all packages missing from the application. The next step would be to import the package into your dart file. Support for this can be found in the package README file in the documentation. Each library has been developed to perform a specific task; each package will require a particular instruction to be followed. It is essential you demonstrate your understanding of the package you are including in your projects, you should evidence this using comments – simply adding a package to your coursework without evidence of understanding will not be tolerated.
