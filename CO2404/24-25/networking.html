Postman

You know from previous labs that in Flutter we need to include the 'http' library, this gives our applications the ability to make HTTP requests. Now we have an additional component to handle, we have the Flutter application and the HTTP request/response. When you build applications to parse HTTP data you need to think about where errors might occurs. Of course you might have errors in your code when parsing objects, but you now might also have errors on the server (backend). Debugging these can be an issue. Therefore, its always best practice to use other tools to debug where the issue might be coming from. For any project that requires an API, I use Postman for creating, sharing and testing an API.

Postman allows you to create and save HTTP/s requests as well as the responses, you set up your connections, input your test criteria, debug the response and save for later. This enables you to visualise the data in a meaningful way. Which results in a more efficient and less tedious way of working.

Of course there are other programs on the market, but I've always found Postman to do an excellent job. Postman works across platforms on:

  • Windows x32

  • Windows x64

  • Linux

  • MacOS

  • Web

Further Reading

As Dart is a single-threaded language, this means that Flutter apps can only do one thing at once. However, this doesn’t mean that Flutter apps have to wait until one task is finished. Flutter apps use an event loop (like Android/Java, iOS/ObjC/Swift etc even JavaScript has an event looper. An event loop is something that runs in the background indefinitely and is called upon periodically – waking up the vent queue for any tasks that need to be run. These events are then actioned, and the event loop places them in the run stack only if the CPU is idle.

Certain tasks slow down this event loop. Reading from a file is quite an extensive process. Writing to a file is even slower. So, you should think about what you are doing and when it's needed. However, don’t worry too much about this as these languages are very smart. The moment an activity knows you are wanting to perform something quite extensive in terms of resources. The event loop moves it out of the CPU queue, a listener is then created which monitors the activity and raises an alert when it is finished waiting. A reference to that listener is returned to the main thread (this is known as a Future). The main thread continues with its process and when the waiting activity is finally resolved, the event loop sees it and runs an associated method (a callback function) on the main thread to handle finishing up the slow event. It is very simple to add this to our code, in fact, you’ve already done it in this lab (Figure 4). In this example our Future is of type String this tells us that when it’s ready we will have a string to deal with. When you have a Future object, you may not have the data, but you have a promise to get that data in the Future.

To get data back from the Future, you are essentially responding to the call when the event fired is ready. For example, myFuture.then(myCallback), this .then() function is how you register that callback function. The callback function should be written to handle the promised data.

We use a void function because there is no way that the .then function can receive a returned value. This makes sense as remember the callback function is no longer running in the main thread. So, it has no way of retrieving the data back into it. Luckily, there are several methods to get a value from the callback. The most understandable is that you use a variable that is defined outside of the callback.

There is another method to get data which is more straightforward. The await call is used in place of the .then. Awaiting pauses the running code to wait for the Future to resolve. However, when you use await inside a function, that function can now cause a blockage to the main thread. You will need to make the call an async one.

There is a lot to understand here, but you will need to continue this learning by reading further about awaits, .then() and async https://dart.dev/codelabs/async-await.
